# StyleCloset

Personal single-user wardrobe app: Vite + React + TypeScript, backed by Supabase (Postgres + Auth + Storage + Edge Functions).

## Deployment topology

- **GitHub**: `AslanBuilt/project-bolt-sb1-rd1ar2yp` (private). This app lives in the `project/` subfolder of that repo, not the repo root. Pushing to `main` auto-deploys to Vercel via its Git integration.
- **Vercel**: project `style-closet` (scope `aiden-ricketts`). Root Directory is set to `project`. Deployment protection is disabled, so the production URL is publicly reachable.
- **Database**: Supabase project `supabase-red-xylophone` (ref `hrprbiregnrbrdnhetht`), provisioned through the Vercel Marketplace Supabase integration — not a standalone Supabase project. This *replaced* an earlier standalone Supabase project (ref `bhxiwpztitzgzslzxkiz`); that old project's data was not migrated over, only the schema (via the migration files in `supabase/migrations/`).
- **Edge Functions**: `ai-tag-item`, `analyze-inspiration`, `outfit-recommend` under `supabase/functions/` run on Supabase's infrastructure, deployed separately from the Vercel frontend build.

## Edge Functions require the anon key as a Bearer token

All 3 Edge Functions are deployed with `verify_jwt: true` (the Supabase default) — every `fetch()` call to them from the frontend **must** include `Authorization: Bearer ${VITE_SUPABASE_ANON_KEY}`, or Supabase rejects the request with `401` before it ever reaches the function code. `TodayPage.tsx` did this correctly from the start; `InspirationPage.tsx` and `AddItemPage.tsx` originally didn't, which silently broke AI inspiration analysis and AI auto-tagging entirely (they weren't hitting a real Gemini rate limit at all — every request was being rejected at the auth layer). Fixed 2026-07-21. If a new Edge Function call site is ever added, make sure it includes this header.

## Color-wheel pairing score (rule-based pass only)

`recommendations.ts`'s `colorsWorkTogether()`/`calculateColorScore()` (used inside `generateOutfitCandidates`, and directly by `TodayPage.tsx` for outfit `colorScore` display) now use real hue-distance color-wheel math instead of the original hardcoded harmony/clash tables.

- **Architecture note, since the build spec named two functions**: the spec asked to add this "alongside the rotation score and inspiration-match bonus in `filterAndScoreItems`" - but `filterAndScoreItems` scores each item alone within its category, with no visibility into what it'll be paired with, so a pairwise hue comparison can't live there. There was already a pairwise color mechanism one layer up, in `generateOutfitCandidates` (`calculateColorScore`/`colorsWorkTogether`) — doing the same underlying job ("do these two colors work together") with a cruder set-based heuristic. Rather than run two overlapping color signals, that existing mechanism's *internals* were upgraded to real color-wheel math, keeping its exact function signature so every call site kept working unchanged. `filterAndScoreItems`'s rotation score and inspiration bonus were not touched.
- **Logic**: `NEUTRAL_COLORS = [black, white, gray, navy, beige, denim]` (exactly the spec's list, not an expanded "feels neutral" set) skip hue math entirely and get a flat bonus. For two chromatic colors, hue distance (via a hardcoded color-name→hue lookup, `COLOR_HUE_MAP`) is bucketed: ≤15° near-identical, ≤45° analogous, ≤100° unrelated, ≤140° triadic, ≤165° split-complementary, >165° complementary. Bonuses are intentionally close together (55-95 range, same overall magnitude as the code it replaced) since this is a soft nudge, never a hard filter.
- **`secondary_color` is considered too** (`bestColorMatch()`): for a pair of items, every combination of {item A's primary/secondary} × {item B's primary/secondary} is scored and the *best* relationship wins — e.g. a yellow top with a blue secondary accent scores based on the blue/orange complementary relationship (88) against orange pants, not the weaker yellow/orange analogous one (70), if that's the stronger pairing.
- **Fail-safe**: an unmapped color name (not in `COLOR_HUE_MAP` and not a neutral — e.g. `"multi-color"`) never errors or excludes the item; it gets a small neutral-adjacent default score (60).
- **Not done this round, deliberately**: the AI (Gemini) pass's prompt wasn't touched — it already reasons about color to some extent on its own; only the rule-based fallback got this upgrade, per spec.
- **Verified**: ran the actual `colorsWorkTogether()` live, comparing new output against the exact old hardcoded results it replaced (captured before deletion, not reconstructed from memory) — e.g. blue+orange rose from a flat hardcoded 60 to 88 (correctly recognized as genuinely complementary, ~180° apart); red+pink rose from a hardcoded "clashing" 20 to 70 (real hue math puts them only ~30° apart - analogous, not clashing). Also verified the real 3-item closet outfit scores 85, and that the `secondary_color` mechanism picks the stronger of two possible pairings correctly.

## Closet Insights and the "archive" feature

Pure read-only stats + a nudge banner, zero AI cost — no new migration was needed at all, because everything it needed already existed:

- **"Archive" was already fully built, under the name `retired`.** Before writing any code for this feature, checked `ClosetPage.tsx` and found: a reversible per-item toggle (`toggleRetired`), already excluded from the closet grid by default (with a separate "Show Retired" view to see them), already excluded from the recommendation candidate pool (`TodayPage.tsx` fetches `.eq('retired', false)`), already excluded from item counts. Only the **UI text** was relabeled to "Archive"/"Archived"/"Unarchive" to match this spec's terminology — the underlying column, variable, and function names (`retired`, `showRetired`, `toggleRetired`) were deliberately left alone to avoid an unnecessary rename touching working code across multiple files for a purely cosmetic ask.
- **Wear count and last-worn were already tracked too**: `clothing_items.times_worn` and `last_worn_date`, maintained by the existing `increment_times_worn()` Postgres function (called once per item whenever `TodayPage.tsx`'s `wearOutfit()` confirms an outfit). No need to derive wear count from outfit history — the running counter was already reliable.
- **`InsightsPage.tsx`** (`/insights`, linked from a card in Settings, not a 7th bottom-nav icon — the nav bar already has 6 items): total + category/color breakdown, top/bottom 5 by `times_worn`, wardrobe % worn (Whering "Spotify Wrapped" framing), and a "haven't worn in N+ days" list with a threshold selector (14/30/60/90, default 30). All computed client-side from `clothing_items` where `retired = false` — no new table or column.
- **Nudge banner** on `TodayPage.tsx`: derived from the same already-fetched, already-`retired=false`-filtered `items` state, so no extra query. Dismissible via `localStorage` keyed by today's date, so it reappears the next day rather than being dismissed forever after one tap — deliberately not "gone until the underlying count changes" like the Inspiration "Review Unconfirmed" banner's pattern, since this spec explicitly asked for dismissible.
- **Verified**: ran the actual stats computation (extracted from `InsightsPage.tsx`) against the real 4-item closet (all worn within the last day right now, so 100% worn and an empty "not worn" list — correct, not a bug) and again with 2 synthetic stale items added, confirming `percentWorn` drops correctly and both a never-worn and a 45-day-old item are correctly surfaced and ranked.

## Weather-aware outfit filtering

Filters/scores recommendations by today's actual weather, at $0 cost (Open-Meteo, no key, free tier is 10k calls/day for ~1/day of usage).

- **Weather source was already partially integrated** (`src/lib/weather.ts` already called Open-Meteo) — but via live browser geolocation on every load, not a stored location. Changed `getCurrentWeather()` to take `(lat, lon)` params instead; added `geocodeLocation()` (Open-Meteo's free geocoding API, same provider, no key) so a typed city name in Settings resolves once to coordinates and gets stored, rather than prompting for device location every visit.
- **Storage**: `style_preferences.location_lat/location_lon/location_name` (resolved once via geocoding) and `temp_offset_f` (personal "runs cold/hot" calibration, default 0) — set in Settings. `clothing_items.warmth_min_f/warmth_max_f` — a rough °F range each item is appropriate for.
- **Warmth is inferred in code, not by AI** (`src/lib/warmth.ts`, category+subcategory keyword lookup) — deliberately, to avoid adding load to the already-tight shared Gemini quota (see below). Computed automatically in `AddItemPage.tsx` at save time for every new item; existing items were backfilled once via a script using the exact same lookup function, so there's no drift between the two.
- **Filtering happens once, upstream of both recommendation passes — not duplicated per-path.** This was a deliberate structural choice to avoid repeating the exact bug class already found with inspiration data (which only reached one of the two paths before that was fixed). The hard filter runs on `eligibleItems` in `TodayPage.tsx`'s `generateOutfits()`, *before* it's handed to either the rule-based scorer (`filterAndScoreItems`) or the AI pass (`outfit-recommend`) — both simply receive a pre-filtered pool and need zero weather-specific code of their own. `outfit-recommend`'s Gemini prompt still gets a weather description, but purely for narrative flavor in the "reason" text — it is not a second filtering point.
- **Hard filter, not a soft score**: `feltTemp = weather.feelsLike + temp_offset_f`; any item whose `[warmth_min_f, warmth_max_f]` doesn't cover `feltTemp` is dropped outright (mirrors Acloset's approach). Items with no warmth data fail open (kept, not excluded) rather than silently vanishing.
- **Precipitation**: `isRainy` prefers Open-Meteo's daily `precipitation_probability_max > 50%` (the spec's explicit threshold) and falls back to a WMO weather-code check if that's unavailable. No water-resistant-shoe exclusion was built — no item currently has a water-resistance tag and none of the existing tagging flows populate one, so building that exclusion now would be inert. The existing `getWeatherRecommendation()` hints banner already surfaces a rain advisory ("Bring an umbrella or waterproof layer") — that already satisfies the "simple note" fallback the spec explicitly allowed instead.
- **Failure handling**: if no location is set, or the fetch fails, `weather` stays `null` and the entire filter block is skipped (guarded by `if (weather)`) — falls through to the existing rotation/inspiration logic unaffected, never blocks a recommendation.
- **Verification**: logged when the filter changes the pool (`console.log` in `generateOutfits()` — felt temp, count excluded, which items and why). Live-tested by running the actual filter predicate against the real 4 closet items and their real backfilled warmth bands across several temperatures (no Gemini involved, so unaffected by the daily quota): at 28°F every item was excluded (this wardrobe currently owns nothing warm enough — a real, useful signal); at 50°F only jeans+sneakers survived; at 70°F everything survived; at 100°F only the jeans got excluded.

## Outfit try-on feature

Visualizes the day's recommended outfit on the user's own photo, generated automatically in the background.

- **Switched to Gemini image generation (2026-07-24), replacing the entire Hugging Face CatVTON/IDM-VTON chain.** The old free `@gradio/client`-based chain (documented in git history if ever needed) is deleted, not disabled — `HF_TOKEN`, `CATVTON_ENABLED`, `callCatVTON`/`callIdmVton`/`tryStep` are all gone. `supabase/functions/outfit-tryon/index.ts` now makes a **single** call to `gemini-2.5-flash-image` (`generateContent`, model confirmed live via the `models.list` endpoint — don't trust doc-page summaries for this API, they've been observed to hallucinate a different endpoint/model name) with the base photo plus every garment reference photo sent together in one request (`contents[0].parts`: one text prompt + N `inlineData` images), `generationConfig: { responseModalities: ["TEXT", "IMAGE"] }`. This removes the old sequential per-step dependency where a bad upper-body result would corrupt the chained lower-body step.
- **Separate paid API key, separate project, deliberately isolated from the free-tier key.** `GEMINI_IMAGE_API_KEY` (Supabase secret) is a different Google AI Studio project ("Closet Generation", Tier 1 Prepay, $10 prepaid balance) from `GEMINI_API_KEY`. `ai-tag-item`, `analyze-inspiration`, and `outfit-recommend` are untouched and keep using `GEMINI_API_KEY` on the free tier. Cost is ~$0.039/image (confirmed live via `usageMetadata` token counts matching Google's pricing page exactly) — no automatic retries were added on top of that; a failed generation just reports `failed`, and the manual Regenerate button (below) is the only way to try again, kept deliberately manual rather than auto-looping given real per-image cost.
- **Shoes are now included, and it works well.** Unlike CatVTON/IDM-VTON's hard `cloth_type: upper|lower|overall` restriction, Gemini's image model isn't restricted to fixed garment slots, so `getTryOnStepItems()` (`TodayPage.tsx`) now also includes the outfit's `shoes` item when present, sent as a third reference image alongside upper/lower. **Verified with real garment photos, not assumed**: a live test with the user's actual white sneakers reference photo produced a render with correctly matched color, silhouette, and sole — confirmed via direct side-by-side comparison against the reference photo, not just a "looked plausible" judgment call. (There's no separate "accessory" category in this app's schema to test — `shirts`/`sweatshirt_jacket`/`pants`/`shorts`/`shoes` is the full category set.) If a future real-world result shows poor shoe fidelity, it's fine to drop shoes back out of `getTryOnStepItems()` and revert to upper+lower only — this isn't a one-way door.
- **Caching, keyed by item combination, not by outfit row** (unchanged from before): cache key (`combo_key`) is the sorted, joined ids of the upper+lower(+shoes) items being visualized. Tracked in the `tryon_results` table (`user_id, combo_key` composite PK; `status`: `generating`/`done`/`failed`; `image_url`). `failed_step` and `skipped` columns still exist in the table but are always written as `null`/`[]` now — there's no more per-step chaining, so a generation either fully succeeds or fully fails as one atomic call; the columns were left in place rather than migrated out since they're harmless unused, not worth a schema change for.
- **Durable completion** (unchanged): the Edge Function's service-role client writes the final `tryon_results` row itself once generation finishes, so a result completes correctly even if the browser that triggered it has since navigated away. The client only polls every 6s for a snappier UI update while still on-screen.
- **Failure handling, including the $10 balance hitting $0**: Google's billing docs say the API stops serving immediately at a $0 balance. The Edge Function's catch block inspects the error for `429`/`403`/"quota"/"billing"/"resource_exhausted" and returns a clear "temporarily unavailable" message in that case instead of a generic failure — written defensively based on Gemini's documented error shapes, but **not yet exercised against a real drained balance** (deliberately spending the $10 to $0 just to test it wasn't done); if the real error shape ever differs, it still falls through to a non-crashing generic failure message.
- **Manual "Regenerate" action** (kept from the prior version, `TodayPage.tsx`): visible directly under every rendered try-on result, since a `done` status still doesn't guarantee the image is accurate — occasional visual wrongness from the generative model itself is expected and not treated as a code bug. Calls `ensureTryOn(items, force=true)`, bypassing the cache check and overwriting the old result via the same upsert.
- **Verified live, end-to-end, through the actual deployed pipeline** — not a standalone script reimplementing the logic. After deploying, a real request was sent straight to the deployed `outfit-tryon` function using the app's real user id, real base photo, and real garment photos (white polo, navy chino shorts, white sneakers); the response, the storage upload, and the `tryon_results` row were all inspected directly, and the resulting image was viewed and visually confirmed to show the same person/pose/background wearing all three correct garments with accurate color and detail (heather-texture polo with its actual placket collar, correct navy shorts, correct white sneaker silhouette) — a materially higher-fidelity result than IDM-VTON ever produced in this app.

## Outfit recommendation logic: inspiration profile and recency awareness

Two gaps fixed 2026-07-22, found by walking the actual code rather than assuming:

- **Confirmed inspiration data now reaches both recommendation passes, not just the AI one.** `inspirationProfile` (aggregated from `inspiration_images` where `confirmed: true`) used to only be fetched inside the AI (Gemini) pass in `TodayPage.tsx`'s `generateOutfits()`. It's now fetched once, before the rule-based Pass 1 runs, and passed into `filterAndScoreItems`/`generateOutfitCandidates` (`recommendations.ts`) as well as the AI pass — `RecommendationContext.inspirationProfile`. In the rule-based scorer, an item gets `+15` if its `primary_color` matches (exact or fuzzy substring, e.g. item color "blue" matches inspiration color "light blue") any confirmed inspiration color, and `+10` if its `pattern` matches any confirmed pattern trend the same way. **Vocabulary caveat**: `item.pattern` is the controlled enum `solid/striped/plaid/floral/geometric/printed/other` — inspiration `patternTrends` is free-text from Gemini and often contains fabric/texture words like "denim" or "ribbed knit" that will never match that enum. The pattern bonus only fires when the two vocabularies happen to overlap (e.g. both say "striped") — this is a real, permanent limitation of comparing a controlled enum against free text, not a bug.
- **The AI (Gemini) pass now receives recency data.** Previously the items sent to `outfit-recommend` carried no `last_worn_date`, and no `recentlyWornIds` were sent either — the AI prompt had zero signal about what was worn recently and could recommend the identical outfit day after day with nothing discouraging it. Fixed: `TodayPage.tsx` now sends `last_worn_date` per item plus `recentlyWornIds` in the request body; `outfit-recommend/index.ts` formats each item with `formatRecency()` (`"last worn Nd ago"` or `"never worn"`, flagged `"— WORN RECENTLY, avoid if a suitable alternative exists"`) and a new prompt rule instructs Gemini to avoid recently-worn items when an alternative exists in the same category.
- **Wardrobe-size caveat, not a bug**: with only one item in a category (e.g. one shirt, one pair of shoes), no rotation logic — rule-based or AI — can vary that category. Repetition there is expected and correct until more items are added to that category.
- Verified live: the rule-based fix was confirmed by directly executing the real `filterAndScoreItems` (via `npx tsx`, no server involved) — without an inspiration profile, three test shirts tied at an identical score; with it, they diverged by the color/pattern bonuses exactly as designed, changing the ranking. The AI-pass fix was confirmed by extracting and running the actual `formatRecency()`/prompt-building logic (not the full Gemini round-trip — the shared Gemini free-tier **daily** quota, see below, was exhausted from same-session testing) — the literal prompt text was confirmed to carry the `WORN RECENTLY` flag and the new rule correctly.

## Fixed bug: silent empty analysis from Gemini MAX_TOKENS truncation

`ai-tag-item` (256 tokens) and `analyze-inspiration` (400 tokens) had `maxOutputTokens` set too low for `gemini-2.5-flash`, which spends part of that same budget on internal "thinking" tokens before generating any visible output. Confirmed via logs (2026-07-22): Gemini returned `finishReason: MAX_TOKENS` with the JSON cut off mid-string (e.g. `{"colorPalette": ["cream",`), `JSON.parse` threw, and the existing catch block correctly fell back to empty defaults — **but the function still returned `success` and the row still got marked `analyzed: true`**, so photos silently ended up with no color/silhouette/pattern data and no visible error anywhere. Fixed by setting `thinkingConfig: { thinkingBudget: 0 }` (these are simple classification/extraction calls, not tasks needing multi-step reasoning) and raising `maxOutputTokens` to 1024 on both.
- **Any inspiration photos analyzed before this fix may still have empty analysis fields** despite `analyzed: true` — use the "Re-run analysis" button (available on any photo regardless of status, added 2026-07-22) to reprocess them now that the token limit is fixed.
- **Correction (2026-07-23): `outfit-recommend` was affected too, contrary to what this note originally said.** At the time, `outfit-recommend`'s 2048-token budget looked sufficient against a 4-item wardrobe with no evidence of truncation. Once the wardrobe grew to 17 items (9 shirts, 0 pants, 7 shorts, 1 shoe), the prompt and Gemini's expected JSON response both grew enough that the same `MAX_TOKENS` truncation started happening here too — confirmed via logs (`"itemIds": ["...", "8db1cf18-3c50-4cab-913c-7c366ed6e4` cut off mid-UUID). This one was worse than the silent-empty-fields case: `JSON.parse` failing caused a fallback to the Edge Function's own bare-bones internal `generateRuleBasedOutfits()` (not the sophisticated client-side `recommendations.ts` engine) — which with 0 pants in the wardrobe only had one branch that could fire, producing exactly one recommendation every time regardless of what the user asked for, and which only uses `activityText` for the reason *string*, never to influence item selection. This looked like "the app only ever returns one outfit" and "my activity/color preference text is ignored" — both were downstream symptoms of this truncation, not separate bugs or by-design limits. Fixed the same way: `thinkingConfig: { thinkingBudget: 0 }` + `maxOutputTokens` raised to 4096 (higher than the other two, since this prompt scales with wardrobe size and will keep growing). **Lesson: a token-limit fix judged sufficient at one wardrobe size can silently regress as the wardrobe grows — if outfit recommendations start looking wrong/generic/single-option again, check for `MAX_TOKENS`/parse-failure logs here before assuming it's a prompt logic or wiring problem.**

## Known limitation: Gemini free-tier rate limit

The `GEMINI_API_KEY` currently in use is on Google's **free tier**, which caps `gemini-2.5-flash` at **5 requests/minute per project** — shared across all 3 Edge Functions, since they all call the same model. Uploading several inspiration photos in one batch (or using multiple AI features close together) can exhaust this in seconds; Google returns `429 RESOURCE_EXHAUSTED` with a `retryDelay`.

**There is also a separate, harder daily cap** (confirmed via logs, 2026-07-22): `GenerateRequestsPerDayPerProjectPerModel-FreeTier`, **20 requests/day**, also shared across all 3 functions on `gemini-2.5-flash`. Unlike the per-minute limit, this one does not reset in seconds — once exhausted, every Gemini call fails until the next day regardless of spacing/retries, and `outfit-recommend` silently falls back to its internal rule-based logic (`source: "rule-based"` in the response) rather than surfacing this distinctly. If live-testing AI behavior and every call unexpectedly returns `source: "rule-based"` even with good spacing, check function logs for this specific quota message before assuming a code bug — heavy same-day testing (including agent-driven testing, not just real usage) can burn through this in a single session.

- `ai-tag-item` and `analyze-inspiration` propagate this as `429` + `retryAfter` (seconds) to the frontend instead of a generic 500. `InspirationPage.tsx` waits that long and retries once before giving up.
- `outfit-recommend` already falls back to a rule-based recommendation on any Gemini failure, so it needed no change.
- This makes normal usage resilient to brief rate limiting, but does **not** raise the actual 5/min ceiling. The real fix is enabling billing on the Gemini key's Google Cloud project (or swapping in a paid-tier key) — set the new key via `supabase secrets set GEMINI_API_KEY=... --project-ref hrprbiregnrbrdnhetht`.

## Environment variables and secrets

Two separate stores, because two separate platforms execute this code:

- **Frontend (Vercel project env vars)**: `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`. These get baked into the client JS bundle at build time — treat them as public, not secret. Managed via `vercel env ls` / `vercel env add`.
- **Edge Functions (Supabase secrets)**: `GEMINI_API_KEY` (Google Gemini API, used by all 3 functions). Set via `supabase secrets set GEMINI_API_KEY=... --project-ref hrprbiregnrbrdnhetht`. **Setting this in Vercel does nothing for the Edge Functions** — they don't run on Vercel and never see its env vars.
- **Local dev**: copy `.env.example`-style values into a local `.env` (gitignored). Never commit real secrets; never hardcode them in source.

Never hardcode credentials in source files. The one existing exception is intentional: `src/contexts/AuthContext.tsx` signs into a single fixed account (`aiden@stylecloset.internal`) with a hardcoded password, because this is deliberately a single-tenant personal app (see `AuthPage.tsx`). That password ships in the public JS bundle regardless of where it's stored, so the deployed URL itself is the real access boundary — don't "fix" this without checking with whoever owns the app first, since it's a design choice, not an oversight.

## Working with the database

Supabase CLI isn't installed globally on this machine (winget doesn't have it, global npm install is deprecated by Supabase) — use `npx -y supabase@latest <command>` for everything.

- **Schema changes**: add a new file to `supabase/migrations/`, then push it:
  ```
  npx supabase@latest db push --db-url "<direct/non-pooling Postgres URL, port 5432>" --yes
  ```
  Get the connection string via `vercel env pull` (look for `POSTGRES_URL_NON_POOLING`) or the Supabase dashboard. Use the direct connection, not the pgbouncer pooled one, for DDL.
- **Storage buckets** are not created by SQL migrations even when RLS policies reference them — create via the Storage Management API (`POST https://<ref>.supabase.co/storage/v1/bucket`, service role key) if a new one is ever needed. Current bucket: `clothing-photos` (private).
- **Edge Functions redeploy**:
  ```
  $env:SUPABASE_ACCESS_TOKEN = "<personal access token from supabase.com/dashboard/account/tokens>"
  npx supabase@latest functions deploy --project-ref hrprbiregnrbrdnhetht --use-api
  ```
  This is a separate auth flow from the DB connection string above — it talks to Supabase's Management API and needs a Personal Access Token, not the DB password.

## Deploying

Normal path: commit, push to `main`, Vercel builds and deploys automatically. No manual `vercel deploy` needed for routine changes. Re-run the Vercel/Supabase CLI steps above only when provisioning changes (new env vars, new migrations, new/changed Edge Functions) — the frontend deploy itself is fully automatic on push.
